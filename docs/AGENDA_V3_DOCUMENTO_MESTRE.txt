0. OBJETIVO DESTE DOCUMENTO
============================================================

Este é o documento mestre da Agenda V3.

Este documento deve ser usado junto com as imagens de referência visual aprovadas (pasta referencias-visuais/).

O objetivo é orientar a criação do app de forma simples, organizada e progressiva.

Regra principal:
- este documento define a base oficial;
- as imagens definem a direção visual;
- pedidos novos do usuário prevalecem sobre o documento, mas devem ser avaliados antes de implementar.

Se o usuário pedir algo fora deste documento, o Claude deve:
1. avisar que o pedido está fora do escopo atual;
2. explicar o impacto no produto, nos dados, no fluxo ou no visual;
3. perguntar se o usuário quer oficializar essa alteração;
4. sugerir a melhor forma de implementar mantendo a organização do projeto;
5. só implementar depois da confirmação.

============================================================
1. O QUE É O APP
============================================================

A Agenda V3 é um aplicativo mobile-first de agenda profissional para autônomos que trabalham com horários marcados.

A ideia central é:

"Uma agenda de papel no celular, moderna, rápida e organizada."

O app deve ajudar o profissional a:

- organizar horários;
- criar agendamentos;
- marcar atendimentos como realizados;
- registrar pagamentos simples;
- controlar pendentes;
- acompanhar clientes;
- consultar relatório de faturamento;
- compartilhar horários pelo WhatsApp;
- configurar serviços, formas de pagamento e bloqueios fixos.

O app nasce inspirado no uso real de barbeiros, mas não deve ser limitado somente a barbeiros.

Público-alvo:
- barbeiro;
- cabeleireira;
- manicure;
- designer de sobrancelha;
- maquiadora;
- tatuador;
- personal;
- terapeuta;
- consultor;
- profissional autônomo com agenda própria.

Princípio do produto:
O app deve ajudar o profissional, não mandar nele.

O usuário precisa conseguir fazer encaixes, ajustes e decisões manuais sem o sistema travar o fluxo.

============================================================
2. PRINCÍPIOS DO PRODUTO
============================================================

1. Simplicidade
Ações comuns devem ter poucos cliques.

2. Velocidade
Agendar, realizar, registrar pagamento e consultar informações deve ser rápido.

3. Controle manual
A agenda não deve empurrar horários automaticamente.

4. Mobile-first
A experiência principal é no celular.

5. Visual profissional
O app deve parecer produto pago, mas sem ficar pesado.

6. Organização
O código, os dados e as telas devem ser previsíveis.

7. Evolução por fases
Primeiro visual, depois interações, depois lógica real, depois backend.

============================================================
3. DECISÕES DE PRODUTO JÁ FECHADAS
============================================================

1. Serviço não controla duração.

O serviço serve para identificar o que foi feito, aparecer no histórico e ajudar no relatório.
Ele não empurra horário, não reorganiza a agenda e não define duração automática.

2. Serviço pode ter valor opcional cadastrado, mas não deve preencher automaticamente o valor do atendimento na primeira versão.

O valor real é informado/editado no momento de finalizar o atendimento.

3. A agenda funciona como lista de horários.

Horários livres são gerados pelas configurações de agenda e não precisam ser salvos individualmente.

4. Agendamento e realizado usam modais grandes ou bottom sheets.

Eles podem ocupar boa parte da tela, mas devem continuar simples e objetivos.

5. Ranking, aniversariantes, clientes sem retornar e detalhe do cliente são páginas próprias.

Não devem ser modais pequenos, porque são telas de consulta.

6. Pendentes entram no relatório no dia do atendimento.

O app não deve tratar pendente como fluxo financeiro por data de recebimento.
Quando um pendente é quitado, o atendimento continua pertencendo ao dia em que foi realizado.

7. Excluir cliente significa mover para lixeira.

O histórico de atendimentos não deve sumir.

8. Com assinatura vencida, os dados continuam preservados.

O usuário pode visualizar parte do app e fazer ações básicas permitidas, mas recursos completos ficam bloqueados até renovar.

9. O app deve começar com HTML, CSS, JavaScript puro e localStorage.

Backend, login real, assinatura real e banco de dados entram depois.

============================================================
4. FASES DE DESENVOLVIMENTO
============================================================

FASE 1 — Estrutura e visual estático
Objetivo:
Criar a base visual do app com todas as telas principais usando dados fictícios.

Entra:
- estrutura de pastas;
- HTML das telas;
- CSS organizado;
- design system;
- menu inferior;
- telas estáticas;
- modais visuais;
- tema escuro;
- tema claro preparado;
- cor principal preparada;
- pequenas interações visuais.

Ainda não entra:
- lógica real completa;
- cálculos reais;
- localStorage completo;
- backend;
- login real;
- assinatura real.

FASE 2 — Componentes e interações simples
Objetivo:
Fazer a interface responder visualmente.

Entra:
- abrir e fechar modais;
- alternar tema;
- alternar cor principal;
- selecionar chips;
- navegar entre telas;
- calendário visual;
- ações simuladas.

FASE 3 — Dados reais com localStorage
Objetivo:
Salvar e manipular dados reais no navegador.

Entra:
- clientes;
- serviços;
- formas de pagamento;
- agendamentos;
- realizados;
- pendentes;
- relatório;
- bloqueios fixos;
- WhatsApp;
- configurações;
- backup local.

FASE 4 — Refinamento e testes
Objetivo:
Testar no uso real e corrigir.

Entra:
- testes no celular;
- ajustes visuais;
- correção de bugs;
- organização de código;
- melhoria de fluxo;
- performance.

FASE 5 — Backend e produto comercial
Objetivo:
Migrar para versão comercial.

Entra futuramente:
- login real;
- cadastro real;
- banco de dados;
- assinatura;
- sincronização;
- backup em nuvem;
- termos e privacidade definitivos.

============================================================
5. STACK E REGRAS TÉCNICAS
============================================================

Stack inicial obrigatória:
- HTML;
- CSS;
- JavaScript puro;
- localStorage;
- Git/GitHub;
- teste local ou GitHub Pages.

Organização:
- CSS separado por responsabilidade;
- JavaScript separado por tela ou módulo;
- dados centralizados em storage.js;
- funções utilitárias em utils.js;
- componentes reaproveitáveis quando fizer sentido;
- nomes claros;
- classes para CSS;
- IDs com prefixo js- quando forem usados por JavaScript.

Regra de alteração:
Antes de alterar algo, entender o arquivo e mudar apenas o necessário.

Ao receber uma tarefa, o Claude deve:
1. entender o objetivo;
2. identificar quais arquivos precisam mudar;
3. preservar o que já funciona;
4. alterar o mínimo necessário;
5. explicar onde mexeu;
6. avisar se houver risco ou conflito.

============================================================
7. DESIGN SYSTEM
============================================================

Direção visual:
- moderno;
- escuro por padrão;
- claro também preparado;
- roxo como cor principal padrão;
- limpo;
- profissional;
- mobile-first;
- com poucos cliques;
- com listas funcionais;
- com cards apenas quando ajudam na hierarquia.

As três principais referências visuais são:
1. Agenda
2. Clientes
3. Detalhe do cliente

Todas as outras telas devem parecer continuação natural dessas três.

------------------------------------------------------------
7.1 Temas
------------------------------------------------------------

O app deve ter:

Tema escuro:
- fundo quase preto;
- cards escuros;
- bordas discretas;
- textos claros.

Tema claro:
- fundo claro suave;
- cards brancos ou quase brancos;
- bordas discretas;
- textos escuros.

O tema escuro é a prioridade visual inicial.

------------------------------------------------------------
7.2 Cor principal
------------------------------------------------------------

A cor principal padrão é roxo.

Cores permitidas futuramente:
- roxo;
- azul;
- ciano;
- verde;
- rosa;
- vermelho;
- amarelo/dourado.

A cor principal deve afetar:
- botões principais;
- item ativo do menu;
- dia selecionado;
- chips ativos;
- bordas de destaque;
- gráficos principais;
- elementos selecionados.

A cor principal não deve afetar:
- pago;
- pendente;
- alerta;
- erro;
- excluir;
- sucesso;
- perigo;
- textos principais;
- fundos gerais;
- cards neutros.

5- a cor principal não pode ser igual a alguma outra cor que já é usada no app, por exemplo: botão de exclusão (ação perigosa) provavelmente será vermelho então o vermelho da cor principal deve ser em outro tom.

------------------------------------------------------------
7.3 Tokens visuais iniciais
------------------------------------------------------------

Tema escuro:
- fundo principal: #0B0B10
- fundo secundário: #11111A
- card: #171724
- card elevado/modal: #1E1E2E
- borda discreta: rgba(255,255,255,0.08)
- texto principal: #F4F4F5
- texto secundário: #A1A1AA
- texto apagado: #71717A
- primária roxa: #7C3AED
- primária forte: #8B5CF6
- primária suave: rgba(124,58,237,0.18)
- sucesso/pago: #22C55E
- atenção/pendente: #F59E0B
- perigo/excluir: #EF4444

------------------------------------------------------------
7.4 Componentes principais
------------------------------------------------------------

Componentes necessários:
- header de página;
- menu inferior;
- card de métrica;
- card secundário;
- lista compacta;
- item de agenda;
- badge de status;
- chip;
- botão primário;
- botão secundário;
- botão fantasma;
- botão perigo;
- botão ícone;
- input;
- textarea;
- select;
- modal pequeno;
- modal grande/bottom sheet;
- estado vazio;
- calendário;
- donut;
- gráfico simples.

------------------------------------------------------------
7.5 Hierarquia visual
------------------------------------------------------------

Agenda:
- lista de horários é protagonista.

Clientes:
- ranking e lista são protagonistas;
- aniversariantes e sem retornar são secundários.

Detalhe do cliente:
- ficha do cliente e histórico são protagonistas.

Relatório:
- faturamento, atendimentos e recebimentos são protagonistas.

Pendentes:
- valor total pendente e lista de quem deve são protagonistas.

Serviços:
- lista de serviços é protagonista.

Formas de pagamento:
- lista de formas é protagonista.

Intervalos:
- lista de bloqueios é protagonista.

Configurações:
- seções organizadas são protagonistas.

12. FLUXOS PRINCIPAIS
============================================================

------------------------------------------------------------
12.1 Fluxo: criar agendamento
------------------------------------------------------------

1. Usuário toca em horário livre.
2. Abre modal Horário livre.
3. Usuário toca em Agendar.
4. Abre modal Novo agendamento.
5. Usuário digita nome do cliente.
6. App sugere clientes existentes.
7. Usuário seleciona cliente existente ou mantém nome novo.
8. Usuário seleciona serviços, se quiser.
9. Usuário adiciona observação, se quiser.
10. Usuário salva.
11. App cria registro em agendaV3:agendamentos com status agendado.
12. Agenda atualiza o horário como agendado.

------------------------------------------------------------
12.2 Fluxo: bloquear horário livre
------------------------------------------------------------

1. Usuário toca em horário livre.
2. Abre modal Horário livre.
3. Usuário toca em Bloquear.
4. Abre modal Bloquear horário.
5. Usuário informa nome opcional.
6. Usuário confirma.
7. App cria bloqueio pontual em agendaV3:agendamentos com status bloqueado.
8. Agenda atualiza o horário.

------------------------------------------------------------
12.3 Fluxo: marcar agendamento como realizado
------------------------------------------------------------

1. Usuário toca em horário agendado.
2. Abre modal Horário agendado.
3. Usuário toca em Realizado.
4. Abre modal Finalizar atendimento.
5. Usuário confere/edita cliente e serviços.
6. Usuário informa se foi pago.

Se foi pago:
7. Usuário seleciona uma ou mais formas de pagamento.
8. Usuário informa valor em cada forma.
9. App soma o valor total.
10. App salva status realizado_pago.

Se não foi pago:
7. Usuário informa valor pendente.
8. App salva status realizado_pendente.
9. Atendimento aparece em Pendentes.

Resultado:
- agendaV3:agendamentos é atualizado;
- relatório considera o atendimento no dia realizado;
- pendentes aparece se não pago.

------------------------------------------------------------
12.4 Fluxo: editar realizado
------------------------------------------------------------

1. Usuário toca em horário realizado.
2. Abre modal Horário realizado.
3. Usuário toca em Editar realizado.
4. Abre modal Editar realizado.
5. Usuário altera cliente, serviços, valores ou status.
6. Usuário salva.
7. App atualiza agendaV3:agendamentos.
8. Relatório e pendentes refletem a alteração.

------------------------------------------------------------
12.5 Fluxo: receber pendente
------------------------------------------------------------

1. Usuário entra em Pendentes.
2. Toca em Receber.
3. App leva para a agenda no dia do atendimento.
4. Usuário abre o realizado pendente.
5. Usuário edita realizado e marca como pago.
6. App remove da lista de pendentes.
7. Relatório continua considerando o atendimento no dia original.

------------------------------------------------------------
12.6 Fluxo: criar cliente
------------------------------------------------------------

1. Usuário entra em Clientes.
2. Toca no botão +.
3. Abre modal Novo cliente.
4. Usuário informa nome e dados opcionais.
5. Salva.
6. App cria cliente em agendaV3:clientes.

Também pode ocorrer dentro do Novo agendamento, se o nome digitado não existir.

------------------------------------------------------------
12.7 Fluxo: mover cliente para lixeira
------------------------------------------------------------

1. Usuário abre Detalhe do cliente.
2. Toca em Editar.
3. Toca em Mover para lixeira.
4. Abre confirmação.
5. Usuário confirma.
6. Cliente sai de agendaV3:clientes.
7. Cliente entra em agendaV3:clientesLixeira.
8. Histórico de atendimentos permanece.

------------------------------------------------------------
12.8 Fluxo: cadastrar serviço
------------------------------------------------------------

1. Usuário entra em Serviços.
2. Toca em +.
3. Abre modal Novo serviço.
4. Informa nome e valor opcional.
5. Salva.
6. App cria serviço em agendaV3:servicos.

------------------------------------------------------------
12.9 Fluxo: cadastrar forma de pagamento
------------------------------------------------------------

1. Usuário entra em Formas de pagamento.
2. Toca em +.
3. Abre modal Nova forma de pagamento.
4. Informa nome.
5. Escolhe tipo obrigatório.
6. Informa taxa opcional.
7. Salva.
8. App cria forma em agendaV3:formasPagamento.

------------------------------------------------------------
12.10 Fluxo: cadastrar intervalo fixo
------------------------------------------------------------

1. Usuário entra em Intervalos.
2. Toca em +.
3. Abre modal Novo intervalo.
4. Informa nome.
5. Escolhe dias da semana.
6. Define horário inicial e final.
7. Salva.
8. App cria bloqueio em agendaV3:bloqueiosFixos.
9. Agenda usa esses bloqueios para exibir horários bloqueados.

------------------------------------------------------------
12.11 Fluxo: compartilhar horários WhatsApp
------------------------------------------------------------

1. Usuário toca no botão WhatsApp da Agenda.
2. Abre modal Compartilhar horários.
3. Usuário escolhe dia ou dias.
4. App calcula horários livres com base em:
   - configurações da agenda;
   - agendamentos;
   - realizados;
   - bloqueios pontuais;
   - bloqueios fixos.
5. App gera mensagem.
6. Usuário toca em Enviar no WhatsApp.
7. App abre WhatsApp com a mensagem preenchida.
8. Usuário confirma o envio manualmente.

------------------------------------------------------------
12.12 Fluxo: relatório
------------------------------------------------------------

1. Usuário entra em Relatório.
2. Escolhe Dia, Semana, Mês ou Ano.
3. App lê agendaV3:agendamentos.
4. Filtra atendimentos realizados no período.
5. Calcula:
   - faturamento;
   - atendimentos;
   - ticket médio;
   - recebimentos por tipo;
   - taxas;
   - pendentes.
6. Exibe resultado.

------------------------------------------------------------
12.13 Fluxo: backup
------------------------------------------------------------

Exportar:
1. Usuário entra em Backup.
2. Toca em Exportar.
3. App junta as chaves agendaV3.
4. Gera arquivo JSON.
5. Usuário salva no dispositivo.

Importar:
1. Usuário toca em Importar.
2. Escolhe arquivo.
3. App valida estrutura.
4. Mostra confirmação.
5. Se confirmado, substitui dados atuais pelas informações do backup.
